# Архитектура Telegram Business Monitor Bot

## Обзор системы

Telegram Business Monitor Bot представляет собой асинхронное Python-приложение, построенное на фреймворке aiogram 3.15.0. Система спроектирована для масштабирования, надежности и безопасности при работе с бизнес-сообщениями Telegram.

## Компоненты системы

### 1. Основное приложение (bot.py)

Файл содержит более 3000 строк кода и включает следующие модули:

#### 1.1 Модуль конфигурации
Управляет всеми настройками приложения, включая токены, идентификаторы администраторов и пути к директориям.

#### 1.2 Модуль базы данных
Класс Database инкапсулирует всю логику работы с SQLite базой данных. Реализует паттерн Repository для абстракции доступа к данным.

#### 1.3 Модуль обработки сообщений
Обрабатывает входящие обновления от Telegram API, включая новые сообщения, изменения и удаления.

#### 1.4 Модуль управления пользователями
Управляет регистрацией, аутентификацией и подписками пользователей.

#### 1.5 Модуль администрирования
Предоставляет интерфейс для управления ботом, пользователями и рассылками.

#### 1.6 Модуль медиа-обработки
Скачивает и сохраняет медиафайлы из сообщений с оптимизацией хранения.

### 2. База данных

#### Схема данных

**Таблица users** - хранит информацию о пользователях
- Первичный ключ: user_id
- Индексы: username, subscription_type
- Связи: один ко многим с business_connections, saved_messages

**Таблица business_connections** - связи между ботом и бизнес-аккаунтами
- Первичный ключ: connection_id
- Внешний ключ: user_id
- Индексы: user_id, is_enabled

**Таблица saved_messages** - архив всех сообщений
- Первичный ключ: id (автоинкремент)
- Внешний ключ: user_id
- Составной индекс: (user_id, chat_id, message_id)
- Индексы: connection_id, is_deleted, is_edited

**Таблица exported_chats** - история экспортов
- Первичный ключ: id
- Внешний ключ: user_id
- Индексы: user_id, exported_at

**Таблица notifications** - система уведомлений
- Первичный ключ: id
- Внешний ключ: user_id
- Индексы: user_id, is_read, created_at

**Таблица payments** - история платежей
- Первичный ключ: id
- Внешний ключ: user_id
- Индексы: user_id, status, created_at

**Таблица statistics** - агрегированная статистика
- Первичный ключ: id
- Внешний ключ: user_id
- Индексы: user_id, stat_type, recorded_at

#### Стратегия хранения медиа

Медиафайлы организованы по пользователям для изоляции данных. Каждый файл именуется по схеме:
```
{type}_{timestamp}_{hash}.{extension}
```

Где:
- type: тип медиа (photo, video, document, audio, voice)
- timestamp: YYYYMMDD_HHMMSS
- hash: первые 8 символов MD5 хеша file_id
- extension: оригинальное расширение файла

## Поток данных

### Прием бизнес-сообщений

1. Telegram отправляет обновление через Business API
2. aiogram dispatcher маршрутизирует его в соответствующий handler
3. Handler извлекает connection_id и проверяет его в БД
4. Проверяется активность подписки пользователя
5. Сообщение парсится для извлечения текста и медиа
6. Медиафайлы скачиваются асинхронно
7. Данные сохраняются в БД с транзакцией
8. Обновляется статистика пользователя

### Обработка удаленных сообщений

1. Telegram отправляет BusinessMessagesDeleted update
2. Извлекается connection_id и список message_ids
3. Для каждого ID выполняется поиск в БД
4. Если сообщение найдено, оно помечается как удаленное
5. Формируется уведомление с оригинальным содержимым
6. Уведомление отправляется пользователю
7. Если есть сохраненные медиа, они также отправляются

### Обработка изменений

1. Получается edited_business_message update
2. Извлекается новое содержимое сообщения
3. Ищется оригинальное сообщение в БД
4. Сохраняется оригинальный текст в поле original_text
5. Обновляется текущий текст
6. Устанавливается флаг is_edited
7. Формируется diff уведомление
8. Отправляется пользователю

## Безопасность

### Аутентификация и авторизация

Система использует многоуровневую проверку:

1. **Проверка Telegram ID** - каждый запрос проверяется на принадлежность зарегистрированному пользователю
2. **Проверка подписки** - доступ к функциям зависит от типа подписки
3. **Проверка прав администратора** - админ-функции доступны только ADMIN_ID

### Защита данных

**Изоляция данных пользователей**
Каждый пользователь имеет доступ только к своим данным через user_id в SQL запросах.

**Безопасное хранение файлов**
Медиафайлы хранятся в директориях, привязанных к user_id, что предотвращает несанкционированный доступ.

**Защита от SQL инъекций**
Все запросы используют параметризованные запросы через sqlite3.

**Логирование безопасности**
Все критические операции логируются с указанием user_id и timestamp.

### Ограничения и квоты

**Rate limiting** на уровне обработки:
- Максимум 100 сообщений в секунду от одного пользователя
- Максимум 10 подключений на пользователя
- Максимум 50MB на медиафайл

**Квоты хранилища:**
- Бесплатный аккаунт: 100MB
- Пробный период: 1GB
- Месячная подписка: 10GB
- Годовая подписка: 50GB

## Производительность

### Оптимизации базы данных

**Индексы для быстрого поиска:**
```sql
CREATE INDEX idx_message_lookup 
ON saved_messages(user_id, chat_id, message_id);

CREATE INDEX idx_deleted_messages 
ON saved_messages(user_id, is_deleted, deleted_at);

CREATE INDEX idx_user_connections 
ON business_connections(user_id, is_enabled);
```

**Очистка старых данных:**
Автоматическое удаление сообщений старше 90 дней для бесплатных аккаунтов.

### Асинхронная обработка

Все IO операции выполняются асинхронно:
- Скачивание медиафайлов
- Запись в базу данных (через пул соединений)
- Отправка уведомлений

### Кэширование

**In-memory кэш для:**
- Данных пользователей (TTL: 5 минут)
- Информации о подключениях (TTL: 10 минут)
- Результатов часто используемых запросов

## Масштабируемость

### Горизонтальное масштабирование

Архитектура поддерживает запуск нескольких инстансов бота:
- Shared SQLite БД заменяется на PostgreSQL
- Добавляется Redis для координации между инстансами
- Используется load balancer для распределения нагрузки

### Вертикальное масштабирование

Оптимизация для одного сервера:
- Connection pooling для БД
- Batch processing для массовых операций
- Lazy loading медиафайлов

## Мониторинг и логирование

### Уровни логирования

**INFO** - нормальные операции
- Запуск/остановка бота
- Новые подключения
- Обработанные сообщения

**WARNING** - потенциальные проблемы
- Неизвестные connection_id
- Превышение квот
- Медленные запросы

**ERROR** - ошибки требующие внимания
- Ошибки БД
- Ошибки скачивания медиа
- Ошибки отправки уведомлений

**CRITICAL** - критические сбои
- Недоступность БД
- Недоступность Telegram API
- Переполнение диска

### Метрики

Система собирает следующие метрики:
- Количество активных пользователей
- Среднее время обработки сообщения
- Количество ошибок в единицу времени
- Использование дискового пространства
- Количество обработанных сообщений

## Обработка ошибок

### Стратегии восстановления

**Временные сбои сети:**
- Exponential backoff при повторных попытках
- Максимум 3 повторные попытки
- Timeout 30 секунд на операцию

**Ошибки БД:**
- Автоматический rollback транзакций
- Логирование проблемных запросов
- Уведомление администратора

**Переполнение очереди:**
- Drop oldest messages при переполнении
- Приоритизация важных обновлений
- Уведомление пользователя о потерях

### Graceful degradation

При частичном сбое сервисов:
- Бот продолжает принимать сообщения
- Откладывает проблемные операции в очередь
- Уведомляет пользователей об ограничениях функционала
